## Story

**As a** sistema NutriMed que passa a acompanhar pacientes ao longo do tempo,
**I want** o paciente como entidade cifrada e duas séries temporais de medições (composição corporal e exames) ligadas a ele,
**so that** consultas deixem de ser eventos isolados e a evolução do paciente possa alimentar a dashboard — sem quebrar as consultas antigas e herdando o compliance da fundação (NFR9/NFR10).

## Acceptance Criteria

1. Migration `0005_patients_evolution` cria a tabela **`patient`** (dono = médico via `user_id` FK) com `name_enc`, `phone_enc`, `birth_date_enc`, `goal_enc` cifrados (AES-256-GCM, sufixo `_enc`), `created_at`/`updated_at`.
2. `consultation` ganha coluna **`patient_id uuid` (FK → patient, NULLABLE)** — consultas antigas (sem paciente) continuam válidas; `patient_label_enc` permanece para legado.
3. Tabela **`body_composition`** (N por paciente): `id`, `patient_id` FK, `measured_at timestamptz`, `source_consultation_id uuid` FK nullable, `values_enc text` (blob JSON cifrado), `created_at`.
4. Tabela **`lab_exam`** (N por paciente): mesma forma de `body_composition` (`patient_id`, `measured_at`, `source_consultation_id`, `values_enc`, `created_at`).
5. Índices para leitura de evolução: `idx_*_patient_id` em `patient(user_id)`, `body_composition(patient_id)`, `lab_exam(patient_id)`, `consultation(patient_id)`.
6. Migration aplica do zero e idempotente sobre 0001–0004 (rastreio em `_migrations`); a suíte de schema permanece verde.
7. Teste prova cripto em repouso: um `values_enc`/`name_enc` inserido **não** contém o texto em claro no storage e só é recuperável com a chave (mitiga T7 — teste, não inspeção).

### Padrão de migrations (NÃO desviar)
- Migrations vivem **inline em TS** em `packages/db/src/migrations.ts` (não há `.sql` em disco). Cada entrada `{ name, sql }`, idempotente, rastreada em `_migrations` por `runMigrations`. Ver 0001–0004.
- Colunas sensíveis usam sufixo `_enc` e guardam base64 de AES-256-GCM (`@nutrimed/crypto`).

### Modelo (alvo)
```
patient            id, user_id→app_user, name_enc, phone_enc, birth_date_enc, goal_enc, created_at, updated_at
consultation       + patient_id uuid → patient (NULLABLE)
body_composition   id, patient_id→patient, measured_at, source_consultation_id→consultation (null), values_enc, created_at
lab_exam           id, patient_id→patient, measured_at, source_consultation_id→consultation (null), values_enc, created_at
```
- `values_enc` = `encryptField(JSON.stringify({...}))`. Bioimpedância: `{ peso, massaMuscular, massaGordura, cintura, imc, pgc }`. Exames: `{ ldl, hba1c, insulina }`. Campos opcionais (medição parcial é válida).
- **Idade NÃO é coluna** — derivada de `birth_date_enc` no servidor.

### O que NÃO fazer
- Não criar serviços CRUD aqui (é a 11.2). Esta story é só schema + migration + teste de cripto.
- Não fazer backfill de consultas antigas (FK nullable resolve).
- Não adicionar colunas por marcador (o blob JSON é a decisão — ADR-011).

### Boundary de agentes
- DDL/migration = **@data-engineer**. `git push`/aplicação em ambiente compartilhado = **@devops exclusivo**. Lembrar: mudou migration ⇒ **reiniciar `npm run dev`** (PGlite só aplica no boot).

### Testing
- Foco em AC7 (cripto provada por teste) e AC6 (aplicar do zero + idempotente), no padrão de `packages/db/src/schema.test.ts`.
