## Story

**As a** time de engenharia do NutriMed,
**I want** um monorepo full-stack TypeScript com pipeline de CI (lint, typecheck, test, build) funcionando do zero,
**so that** exista o esqueleto técnico sobre o qual compliance, abstração de fornecedores e todas as features serão construídas com qualidade garantida por automação desde o primeiro commit.

## Acceptance Criteria

1. O monorepo TypeScript compila e os comandos `lint`, `typecheck`, `test` e `build` passam localmente e no CI (ref. Critério de Aceitação 1 do E1).
2. A estrutura de workspaces separa, no mínimo, app frontend (Next.js) e pacote(s) de serviços/domínio Node, com tipos compartilháveis entre eles (ADR-001 — coerência de linguagem, reaproveitamento de tipos).
3. Existe um pipeline de CI (GitHub Actions) que executa `lint → typecheck → test → build` em push e pull request, e falha o check se qualquer etapa falhar.
4. Há pelo menos um teste unitário "smoke" rodando e passando, provando que o runner de testes está configurado de ponta a ponta.
5. Caminhos de import absolutos estão configurados (Constitution Art. VI) — nada de `../../..` em código de produção.
6. `README` (ou doc equivalente) documenta como instalar dependências e rodar cada comando (`dev`, `lint`, `typecheck`, `test`, `build`).
7. Nenhum segredo é versionado; `.env.example` cobre as variáveis necessárias e `.gitignore` cobre `.env`, artefatos de build e `node_modules`.

## Dev Notes

> Contexto suficiente para implementar sem reler a arquitetura. Fontes: `docs/architecture.md` §9, §10, ADR-001; `docs/epics/epic-01-fundacao-compliance.md`; `docs/frontend-spec.md` §11.

### Decisões de arquitetura aplicáveis

- **ADR-001 (Aceito):** Monorepo full-stack TypeScript — Next.js (frontend) + serviços Node no mesmo monorepo. Motivação: coerência de linguagem e reaproveitamento de tipos entre cliente e servidor (`architecture.md` §9, §10).
- **Stack proposta (architecture §9 / frontend-spec §11):** Next.js 16+ / React / TypeScript / Tailwind / shadcn / Zustand / React Query no frontend; Node/TS para serviços de domínio + futuro WS gateway. **Esta story entrega apenas o esqueleto** — não implementa serviços de domínio, WS, nem providers.
- **Escopo desta story (E1 — IN):** "Monorepo TypeScript (Next.js + serviços Node) — ADR-001" e "CI (lint, typecheck, test, build)". As demais entregas do E1 (auth, consent, cripto, auditoria, provider interfaces, disclaimers) são **outras stories** — não fazem parte de 1.1.
- **Art. VI (Constitution) — Absolute Imports:** configurar paths absolutos; evitar imports relativos profundos em código de produção.

### Estrutura de pastas existente (relevante)

- Já existe `.github/` no repo — o workflow de CI vai dentro de `.github/workflows/`.
- Há `.env` e `.env.example` na raiz — reconciliar/estender, **não** versionar segredos reais.
- `docs/` contém todo o planejamento (PRD, arquitetura, épicos) — não tocar.

### O que NÃO fazer nesta story (evitar over-engineering)

- Não implementar `ISttProvider`/`ILlmProvider`/`IVideoAssetProvider`/`IKnowledgeRetriever` (isso é a story de Provider Abstraction Layer do E1).
- Não criar modelo de dados / Postgres / cripto (story de modelo de dados + cripto do E1).
- Não escolher vendor de STT/LLM/Vídeo — isso é decisão de POC, fora do escopo da arquitetura (architecture §9, nota sobre NFR8).
- Manter as interfaces e a estrutura mínimas; não antecipar Fases 2/3.

### Boundary de agentes (autoridade)

- `git push`, criação de PR e configuração de infraestrutura de CI (runners/secrets do GitHub) são **exclusivos do @devops** (ver `.claude/rules/agent-authority.md`). @dev faz `git add`/`commit` locais; ao chegar em push/PR, delega a @devops.

### Testing

- **Localização dos testes:** colocar testes junto ao código (`*.test.ts`/`*.test.tsx`) ou em `__tests__/` por workspace — definir o padrão e documentá-lo no README.
- **Framework:** Vitest (recomendado para o ecossistema Vite/TS) ou Jest — escolher um e padronizar em todos os workspaces.
- **Requisito mínimo desta story:** ≥ 1 teste smoke que passa, provando o runner configurado ponta a ponta (AC 4). Não há lógica de negócio para testar ainda.
- **CI:** o passo `test` do workflow deve executar a suíte e reprovar o check em falha (AC 3).
- **Padrão de qualidade (architecture/Constitution):** lint e typecheck devem passar antes de marcar qualquer task como completa.
