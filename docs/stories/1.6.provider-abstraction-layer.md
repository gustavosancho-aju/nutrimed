## Story

**As a** arquitetura do NutriMed exposta a Supplier Power ALTO e a risco de custo,
**I want** as 4 interfaces de abstração de fornecedores (STT, LLM, Vídeo, Retriever) definidas com ≥ 1 implementação fake testável cada,
**so that** STT/LLM/Vídeo/RAG (E2/E5/E8) se conectem a contratos estáveis, destravando testes e desenvolvimento sem lock-in nem dependência de credenciais de vendor.

## Acceptance Criteria

1. As **4 interfaces** existem como contratos TypeScript: `ISttProvider`, `ILlmProvider`, `IVideoAssetProvider`, `IKnowledgeRetriever` (ref. Critério de Aceitação 6 do E1 / NFR8, ADR-002).
2. Cada interface tem **≥ 1 implementação fake/stub testável**, permitindo desenvolver e testar o domínio sem provider real nem chaves de API.
3. As assinaturas derivam de `architecture.md` §5 (sem inventar capacidades além de NFR8/NFR5/NFR7 — Article IV):
   - `ISttProvider.openStream({ lang: 'pt-BR' })` → sessão que emite segmentos parciais e finais.
   - `ILlmProvider.complete({ system, context, transcript })` → `PersonaContribution`.
   - `IKnowledgeRetriever.retrieve(personaId, query, k)` → `KbChunk[]` (restrito ao namespace da persona — FR21).
   - `IVideoAssetProvider.getClip(personaId, state)` → `ClipRef` (catálogo pré-renderizado: `ouvindo|pensando|sinalizando`).
4. As interfaces vivem numa camada/pacote que o domínio importa, **sem** que o domínio conheça nenhum vendor concreto (desacoplamento NFR8).
5. Os fakes são determinísticos e suficientes para os testes de stories futuras (ex.: fake de STT emite uma sequência fixa de segmentos; fake de LLM retorna uma `PersonaContribution` previsível).
6. As interfaces são **mínimas** — sem antecipar Fase 2 (TTS) ou Fase 3 (avatar interativo); essas entram como novas implementações no futuro, sem refatorar o domínio.

### Contratos de referência (architecture §5 — ilustrativos, derivam de NFR8/NFR5/NFR7)

```typescript
interface ISttProvider {
  // streaming PT-BR; emite segmentos parciais e finais
  openStream(opts: { lang: 'pt-BR' }): SttSession; // -> AsyncIterable<TranscriptSegment>
}

interface ILlmProvider {
  complete(req: { system: string; context: KbChunk[]; transcript: string }):
    Promise<PersonaContribution>;
}

interface IKnowledgeRetriever {
  // recupera SÓ dentro do namespace da persona (FR21)
  retrieve(personaId: PersonaId, query: string, k: number): Promise<KbChunk[]>;
}

interface IVideoAssetProvider {
  // MVP: catálogo pré-renderizado (NÃO streaming em tempo real) [NFR7]
  getClip(personaId: PersonaId, state: VideoState): ClipRef; // ouvindo|pensando|sinalizando
}
```

> **Article IV (No Invention):** os contratos acima são "ilustrativos" na arquitetura. @architect deve mantê-los **mínimos e fiéis** a NFR8/NFR5/NFR7/FR21/ADR-007. Não adicionar métodos especulativos (ex.: TTS, streaming de vídeo, batch) — Fases 2/3 entram como novas implementações depois (AC 6).

### Por que esta story é a espinha dorsal (NFR8 / ADR-002)

- **architecture §5/§14:** "isola o risco de fornecedor (preço/disponibilidade), permite A/B de qualidade, e prepara Fase 2 (TTS) e Fase 3 (avatar) como novas implementações — sem refatorar o domínio."
- **Desbloqueia:** E2 (implementa `ISttProvider` real), E5 (`ILlmProvider` + `IKnowledgeRetriever`), E8 (`IVideoAssetProvider`). Os **fakes** desta story permitem que E3 (walking skeleton) e os motores sejam desenvolvidos/testados **antes** de escolher vendor (decisão de POC).
- **FR21 (escopo por persona):** `IKnowledgeRetriever.retrieve` recebe `personaId` e recupera só do namespace daquela persona — impede o cardiologista de "inventar" endocrinologia (architecture §6).
- **ADR-007:** vídeo é catálogo **pré-renderizado**, não avatar em tempo real — o contrato reflete isso (`getClip`, não `streamAvatar`).

### Tipos a definir (coordenar com 1.3, evitar duplicação — IDS)

- `PersonaContribution`: alinhar com `CONTRIBUTION` de §8 (`type: atencao|sugestao|hipotese|sintese`, `severity: normal|critical`, `text`, etc.) — reusar tipos onde possível.
- `PersonaId`: enum das 3 personas (Aurélio/Paulo/Yara — ver `docs/personas-board.md`).
- `VideoState`: `ouvindo | pensando | sinalizando` (§5).

### O que NÃO fazer

- Não implementar nenhum provider **real** (STT/LLM/Vídeo/RAG vivem em E2/E5/E8).
- Não escolher vendor (decisão de POC — architecture §9).
- Não criar o pipeline de ingestão de KB (E5) — aqui só o contrato `IKnowledgeRetriever` + fake.
- Não antecipar Fase 2/3 (AC 6).

### Boundary de agentes

- Definição de contratos/decisão de design = **@architect**. Implementação de fakes/testes = **@dev**. `git push`/PR = **@devops exclusivo**.

### Testing

- **Foco:** AC 2/5 — fakes determinísticos exercitados contra os contratos. Os fakes são um **ativo reutilizável** por todas as stories de E2–E8 (IDS: serão REUSE).
- **Framework/localização:** padrão da Story 1.1.
- Garantir que o domínio compila importando **apenas** as interfaces (AC 4) — um teste de tipo/compilação serve.
