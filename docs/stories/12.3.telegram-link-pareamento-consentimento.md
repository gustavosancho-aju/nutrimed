## Story

**As a** sistema que precisa ligar um `chat_id` do Telegram ao paciente certo, com consentimento,
**I want** um pacote que gere/valide códigos de pareamento de uso único e exponha um gate de canal (default NEGA),
**so that** o vínculo seja inequívoco, consentido e revogável (ADR-013/ADR-014), espelhando o `@nutrimed/consent`.

## Acceptance Criteria

1. `createPairingCode(db, patientId, createdByUserId, opts?)`: gera um código curto legível, grava **apenas** `code_hash` (SHA-256) + `expires_at` (TTL default 15 min, configurável por `PAIRING_CODE_TTL_MIN`); retorna o **código em claro** (exibido 1x pela UI); auditado.
2. `redeemPairingCode(db, chatId, code)`: valida hash/expiração/consumo; se válido → cria/reativa `telegram_link` (`consent_granted=true`, `linked_at`), marca o código `consumed_at`, retorna `{ok:true, patientId}`; senão `{ok:false, reason:'invalid'|'expired'|'consumed'}`. Código é **uso único**.
3. `resolvePatientByChat(db, chatId)`: retorna `patientId` **somente** se houver vínculo **ativo** (`revoked_at IS NULL`) e consentido; senão `null`.
4. `isChannelAuthorized(db, chatId)`: **gate de servidor** booleano (espelha `isCaptureAuthorized`) — **default NEGA**.
5. `revokeChannel(db, patientId)`: seta `revoked_at`; idempotente; auditado.
6. `getLinkStatus(db, patientId)` → `ChannelConsent { chatId, patientId, granted } | null`.
7. O índice único parcial (12.1) garante ≤1 canal ativo por paciente; **reparear após revogar** funciona.
8. Testes: fluxo feliz (gerar→resgatar→resolver); expirado; consumido (2º resgate falha); inválido; gate **nega** sem consentimento; revogar; re-parear.
