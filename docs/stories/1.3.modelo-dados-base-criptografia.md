## Story

**As a** sistema NutriMed que manipula dados sensíveis de saúde,
**I want** o modelo de dados base (USER, CONSULTATION, CONSENT, AUDIT_LOG) persistido com criptografia em repouso e em trânsito,
**so that** dados clínicos sejam armazenados com segurança jurídica (LGPD/NFR9) desde o primeiro registro, sustentando consentimento e auditoria.

## Acceptance Criteria

1. Existe um schema Postgres com as entidades base **USER, CONSULTATION, CONSENT, AUDIT_LOG** e seus relacionamentos conforme `architecture.md` §8 (ref. Critério de Aceitação 4 do E1).
2. Dados de saúde são criptografados **em repouso** e **em trânsito** (NFR9), de forma **verificável por teste automatizado** (não só por inspeção — mitigação de T7).
3. As migrations são versionadas, reproduzíveis e aplicáveis do zero (idempotência do estado final).
4. A conexão com o banco força TLS/SSL (criptografia em trânsito).
5. Campos sensíveis (dados de saúde / PII) usam criptografia em repouso (coluna/campo cifrado ou cripto no nível de storage), documentada e testada.
6. O modelo respeita as restrições de integridade dos relacionamentos de §8: `USER 1—N CONSULTATION`, `CONSULTATION 1—1 CONSENT`, e a fundação para `AUDIT_LOG` (proveniência detalhada é a Story 1.5).
7. Há um item de follow-up registrado para **residência de dados no Brasil** (LGPD), encaminhado para a Story 1.8 (ADR jurídico) — sem decidir aqui.

### Modelo de dados (architecture §8 — relevante a esta story)

Entidades base e relacionamentos:
- `USER ||--o{ CONSULTATION : conduz`
- `CONSULTATION ||--|| CONSENT : exige`
- `CONTRIBUTION ||--o{ AUDIT_LOG : registra` (CONTRIBUTION/PERSONA/TRANSCRIPT_SEGMENT/CLINICAL_NOTE/KB_CHUNK **não** são desta story — entram em E2/E4/E5/E6/E9)

`AUDIT_LOG` (campos de §8, proveniência completa detalhada na Story 1.5):
- `id uuid`, `contribution_id uuid`, `kb_sources jsonb` ("proveniência — defesa CFM"), `model_version text`, `created_at timestamptz`

> ⚠️ Nesta story criamos a **fundação** de `AUDIT_LOG` (tabela + relacionamento). A lógica de gravação de auditoria com proveniência (gatilho, fontes KB, versão de modelo) é a **Story 1.5**.

### Requisitos NFR aplicáveis

- **NFR9 (architecture §8/§11):** "dados de saúde em repouso e em trânsito" criptografados; minimização; considerar residência BR.
- **NFR10:** auditabilidade — toda contribuição rastreia gatilho/fontes/versão de modelo (sustentado pela tabela criada aqui, lógica na 1.5).
- **T7 (riscos):** "validar criptografia e auditoria por teste automatizado, não só por inspeção" → **AC 2 é teste obrigatório**, não inspeção.

### Dependências e coordenação

- **Depende de 1.1** (monorepo + ferramenta de migration na stack).
- **Coordena com 1.2** (auth): a entidade `USER` é compartilhada — alinhar para não duplicar (IDS). Se 1.2 já definiu `USER` mínimo, esta story estende; senão, esta é a fonte de verdade do schema `USER`.
- **Desbloqueia 1.4** (CONSENT writes) e **1.5** (AUDIT_LOG writes com proveniência).

### O que NÃO fazer

- Não criar entidades de E2+ (TRANSCRIPT_SEGMENT, CONTRIBUTION, KB_CHUNK, etc.).
- Não decidir residência de dados (encaminhar à 1.8).
- Não implementar a lógica de auditoria/consent (apenas a tabela base de AUDIT_LOG).

### Boundary de agentes

- DDL/migrations/RLS = **@data-engineer**. `git push`/PR e aplicação de migrations em ambiente compartilhado = **@devops exclusivo**.

### Testing

- **Foco:** AC 2 e 5 — testes que **provam** a criptografia (dado ilegível em claro no storage; conexão sem TLS recusada). Esse é o coração da story (mitiga T7).
- **Migrations:** testar aplicação do zero (AC 3) — banco limpo → schema final sem erro.
- **Framework:** seguir o padrão da Story 1.1. Testes de DB podem usar container/efêmero local.
- **Nunca** versionar segredos de conexão; usar `.env.example`.
