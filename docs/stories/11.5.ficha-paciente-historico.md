## Story

**As a** nutrólogo abrindo o acompanhamento de um paciente,
**I want** uma ficha com os dados do paciente e o histórico das consultas dele,
**so that** eu veja a trajetória num só lugar e acesse o resumo (nota) de cada consulta — preparando o terreno para a dashboard de evolução.

## Acceptance Criteria

1. `/patients/[id]` mostra os dados do paciente: nome, telefone, data de nascimento + **idade derivada** (`computeAge`), e principal objetivo em destaque.
2. **Validação de posse:** se o paciente não existir ou não pertencer ao médico logado, a ficha não é exibida (404/redirect) — sem vazamento entre médicos.
3. **Histórico de consultas** do paciente (mais recente primeiro): data e status, cada uma como link para `/consultations/[id]` (onde a nota/resumo de E9 já é exibida).
4. Há um **ponto de entrada para a Dashboard de evolução** (`/patients/[id]/dashboard`) — a rota é ativada na Fase 3; aqui o controle indica isso claramente sem gerar erro.
5. `listConsultationsByPatient(db, patientId)` em `@nutrimed/consent` retorna `{ id, status, createdAt }[]` ordenado desc, com teste unitário.
6. Design system existente; navegação por teclado.
