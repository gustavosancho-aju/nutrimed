## Story

**As a** sistema que acompanha o paciente entre consultas pelo Telegram,
**I want** as tabelas de metas nutricionais, registro diário de consumo e vínculo Telegram criadas numa migration versionada e idempotente,
**so that** as camadas superiores (serviços cifrados, gate de consentimento, bot) tenham uma base persistente coerente com o padrão de schema do projeto.

## Acceptance Criteria

1. `packages/db/src/migrations.ts` ganha a entrada `0006_telegram_nutrition` no array `MIGRATIONS`, aplicada por `runMigrations` no boot; **idempotente** (`IF NOT EXISTS`) — reexecutar não reaplica nem falha.
2. Cria `nutrition_goal` (`id` uuid PK `gen_random_uuid()`, `patient_id` FK→`patient(id)`, `set_by_user_id` FK→`app_user(id)`, `effective_from` date, `values_enc` text, `created_at`) + índice `idx_nutrition_goal_patient (patient_id, effective_from DESC)`.
3. Cria `food_log_entry` (`id`, `patient_id` FK, `eaten_at` timestamptz, `source` text default `'telegram'`, `photo_ref` text nullable, `values_enc` text, `model_version` text nullable, `created_at`) + índice `idx_food_log_patient_eaten (patient_id, eaten_at)`.
4. Cria `telegram_link` (`chat_id` text PK, `patient_id` FK, `consent_granted` bool default false, `linked_by_user_id` FK nullable, `linked_at`, `revoked_at`, `created_at`, `updated_at`) + **índice único parcial** `idx_telegram_link_patient_active (patient_id) WHERE revoked_at IS NULL` (1 canal ativo por paciente).
5. Cria `telegram_pairing_code` (`id`, `patient_id` FK, `created_by_user_id` FK, `code_hash` text, `expires_at` timestamptz, `consumed_at` timestamptz nullable, `created_at`) + índice `idx_pairing_code_hash (code_hash)`.
6. Teste de schema (espelha `schema.test.ts`) em PGlite: a migration aplica; tabelas/colunas/índices existem; reexecução é idempotente; o índice único parcial **impede** 2 vínculos ativos para o mesmo paciente mas **permite** reparear após revogar.
